How the Model Actually Runs Right Now

This page explains the current live LYRN chat execution path: how a user message becomes a queued job, how the active job frame is assembled, exactly which layers are injected into the model context and in what order, and how the final answer is captured back into chat history.

Trigger File Driven
Context Layered
Snapshots Job Scoped
Output Separated
Core Principle
The model does not just receive a raw prompt. It receives a structured stack of stable system identity, active world/job framing, live state overlays, bounded job instructions, and then the actual user message as the final user turn.
Execution Shape
watch → build → run → capture

The runner stays idle until triggered, assembles the active context stack, generates, writes raw output, and then hands the result to the capture layer.

Why This Matters

Job instructions, RWI framing, snapshots, and deltas are not all the same thing. Each layer has a separate purpose and should enter the context stack in a controlled order.

Current Chat Rule

The real user text is the final user message. Job instructions are placed as system-side guidance instead of impersonating the user turn.

02 // Injection Order Inside the Model Context
01

Master Prompt / Core System Identity

This is the stable base layer. It defines the model's overall operating identity, baseline behavior, and the non-job-specific rules that should exist before anything dynamic is added.

Purpose: establish the persistent system frame before any task, memory, or runtime overlays are applied.
02

RWI Frame / Active World Interface

This layer is the active task world frame. In practice, it is the job-facing world or interface context that tells the model what space it is operating inside, what kind of loop is active, and what structural affordances matter for this execution.

Purpose: define the working environment for the active loop so the model is not operating in a blank abstract space.
03

Active Snapshots via DSManager

Snapshots are the bounded contextual inserts that are activated for the current job. These can hold job-specific framing, relevant memory slices, or temporary working context. They exist to enrich the active execution without becoming permanent prompt clutter.

Purpose: inject only the currently relevant contextual slices for the live job, then let them drop away when the job ends.
04

Chat History

The runner then adds prior user/assistant turns from the chat history layer. This preserves the immediate conversational thread and gives the model continuity across turns without confusing history with system-level control layers.

Purpose: preserve conversational memory and turn-by-turn continuity.
05

Deltas / Mutable Runtime State

Deltas are the live mutable overlays: current time, running summaries, loose runtime stats, and other state that can change between executions. They come after the larger framing layers because they are lightweight runtime facts, not the foundational identity or world frame.

Purpose: inject fast-changing state without rewriting the stable system layers.
06

Job Instructions

For chat-source jobs, the job instructions are appended as a system-side guidance layer rather than being treated as the user's message. This keeps job logic separate from the human's actual input.

Purpose: tell the active job how to behave without corrupting the meaning of the user turn.
07

Actual User Message

The final layer is the real user text as the true user turn. This is what the model should answer. Everything before it is preparation, framing, or runtime context.

Purpose: ensure the model responds to the person, not to internal orchestration text.
03 // What Each Layer Is Doing
A

System Core

Master prompt and persistent identity rules. This is the durable operating layer that should survive across many different jobs and interfaces.

B

World / RWI Frame

The current operating space. It explains the active interface or loop frame so the model understands what kind of system it is inside during this execution.

C

Snapshot Layer

Selected contextual slices loaded only for the active job. This is where temporary relevance is injected without making the permanent context dirty.

D

Conversation Layer

Recent user/assistant turns from chat history. This gives continuity and lets the model answer in the context of the current exchange.

E

Delta Layer

Live overlays such as runtime state, rolling summaries, time, or other mutable information that changes more often than the snapshot or system layers.

F

Job Logic Layer

Task guidance for the current execution. This controls job behavior, formatting, or routing intent without pretending to be the person speaking.

04 // Why the Order Matters
From Stable To Volatile

Identity before state

The stack moves from the most stable layers to the most volatile. First comes enduring system identity, then world/job framing, then temporary contextual slices, then recent conversation, then live runtime overlays, then current job guidance, and finally the user turn.

This keeps changing information from overwriting the deeper system frame and helps the model interpret short-lived state in the right context.

Job Boundedness

Snapshots and instructions should not linger

Active snapshots and job instructions are meant to be scoped to the current run. They enrich the live job, but they should not remain as stale residue that bleeds into later jobs.

That separation is what keeps one bounded agent loop from polluting the next one.

05 // Input, Output, and Capture Roles
Layer / File Role What it is responsible for
/api/chat Entry point Accepts the user message, writes the structured input payload, creates the chat-processing lock, and queues the first job.
jobs/job_input.json Structured input Stores the real user message plus source metadata and then receives merged job instructions before the runner executes.
trigger_chat_generation() Job merge point Combines the current payload with job instructions, clears stale raw output, and triggers the runner.
model_runner.py Context assembler Reads the structured payload, assembles the message stack in order, runs generation, and writes raw model text to a separate output file.
jobs/job_raw_output.txt Raw model output Holds only the model's raw generated text. It is intentionally separate from the input payload.
chat_watcher_bg.py Capture layer Waits for generation to finish, extracts the response, reads the original user message, writes the chat record, and clears the lock.
chat/chat_*.txt Conversation record Stores the finalized user/model pair that the UI and the chat history layer both read later.

07 // Current Architectural Benefits
Benefit

Clear User Turn

The human's message stays the real user turn instead of being buried under orchestration text.

Benefit

Bounded Context

Snapshots and job instructions can be job-scoped instead of becoming stale baggage.

Benefit

Input / Output Separation

Structured input and raw output live in different files, which makes capture more reliable.

Benefit

Composable Loops

The same runner pattern can support chat, automation, and future job types while keeping the stack ordered.
