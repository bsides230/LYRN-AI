They represent text or other content as high-dimensional vectors. At query time, you convert the input into a vector and compare it to stored vectors to find similar items.\r\n\r\nPostgres with pgvector already supports storing and searching over vectors. But generating and maintaining those embeddings has been left to the application. This often means building a separate pipeline just to keep vector data in sync.\r\n\r\nAutomatic embeddings bring that pipeline into the database. You can manage embedding generation using SQL, triggers, and extensions like pgmq, pg_cron, and pg_net. No new runtimes or services are required.\r\n\r\nThe problem: external embedding pipelines create drift and complexity#\r\nMost teams implementing semantic features in Postgres end up building their own pipeline. The general pattern looks like this:\r\n\r\nStore source content (e.g. documents, tickets, articles)\r\nGenerate an embedding using an external model\r\nStore the result in a vector column\r\nRe-run the embedding job if the content changes\r\nHandle retries if the model fails or times out\r\nThis pipeline is easy to describe but hard to implement. It introduces inconsistency between your source of truth (Postgres) and derived data (the embeddings). It also requires background workers, queues, observability, and external coordination.\r\n\r\nHere are some ways this pipeline can fall apart:\r\n\r\nDrift. If you update the content but forget to re-embed it, your search quality drops.\r\nLatency. Some embedding APIs are slow or rate-limited. If you call them synchronously on write, you add latency to the write path.\r\nLack of resilience. If your background worker dies or the queue fails, you may not notice until things break.\r\nSchema duplication. Your application ends up duplicating logic that could live in the schema.\r\nWhat are automatic embeddings?#\r\nAutomatic embeddings move the vector generation step into Postgres. Not literally. Inference still happens via an external model, but the responsibility for coordinating that process becomes part of your database.\r\n\r\nWhen a row is inserted or updated, Postgres can automatically enqueue a job to generate or refresh its embedding. That job runs in the background, retries if it fails, and stores the result back into the vector column.\r\n\r\nThis approach has a few benefits:\r\n\r\nNo drift. Embeddings stay in sync with content updates.\r\nBring your own model. You can point to any API that returns a vector.\r\nAll SQL. You can enqueue, inspect, and retry embedding jobs without leaving Postgres.\r\nA number of use cases get easier when embeddings are automatically managed:\r\n\r\nBuild semantic search without leaving SQL\r\nKeep embeddings fresh as data changes\r\nUse vector search for deduplication or anomaly detection\r\nCombine structured and semantic filters in a single query\r\nEnrich or classify rows using embedding-based inheritance\r\nDesign patterns for generating embeddings#\r\nThere are two approaches to automatic embeddings today:\r\n\r\nGenerated columns#\r\ncreate table documents (\r\n  id uuid primary key,\r\n  content text,\r\n  embedding vector(1536) generated always as (embed(content)) stored\r\n);\r\n\r\nThis uses a generated column to call an embedding function on write. It only works if your model is local and fast. In practice, this approach with the embed() function blocks the write path and doesn’t scale well.\r\n\r\nTrigger-based asynchronous embeddings#\r\nThis is the pattern we use at Supabase.
